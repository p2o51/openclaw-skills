# Deep Research — OpenClaw 多平台研究技能


## 支持的平台

| 平台 | 内容特性 | 适用场景 |
|------|---------|---------|
| 小红书 | 小众、精致、有态度的生活方式内容；测评和体验帖子质量高，但圈子相对垂直 | 消费决策、小众文化、旅行、摄影、穿搭等垂直话题 |
| B站 | 游戏向内容占主导，同时有高质量的技术/科普/纪录片内容；UP主观点鲜明 | 游戏、动漫、技术向话题；适合找有深度的视频评测 |
| 微博 | 互联网黑话、梗文化、亚文化、厕所笑话；实时舆论场，情绪化内容多 | 追热点、了解某话题的大众情绪和流行梗；内容真实性需打折扣 |
| Twitter/X | 国际视角、开发者/科技圈、英文一手信息；中文圈也有一定规模 | 全球趋势、技术前沿、科技圈动态 |
| 抖音 | 游戏、娱乐、互联网黑话/亚文化；大众向，内容偏快消 | 流行文化、游戏相关、感受大众舆论 |

### 📹 视频内容处理

小红书、B站、抖音的搜索结果中会包含视频链接。如果需要深入了解视频内容（而非仅看标题/简介），应调用 **`getnote-video-summary` skill** 对视频进行转写和总结，再纳入研究结果。

**建议触发场景**：
- B站：UP主发表了重要观点/评测，标题本身信息量不够
- 小红书：视频笔记（非图文），无法从封面和标题判断内容质量
- 抖音：话题讨论类视频，需要了解具体说了什么

研究计划阶段应提前告知用户是否计划调用视频转写，因为这会增加时间。

## 搜索风格

用户可在 prompt 中指定风格，默认为**日常**：

- **日常模式**（默认）：口语化总结，优先中文平台，注重真实用户反馈和体验感
- **学术模式**（用户说「学术」「严肃」「专业」等关键词时切换）：结构化分析，优先 Twitter/X 和 B站专业内容，引用具体数据和来源，语气克制

## 本地留痕系统

每次 research 都必须在工作目录创建一个独立文件夹，保存完整的研究过程和原始素材。

### 文件夹命名规则

```
research_[YYYYMMDD_HHMM]_[话题slug]/
```

例：`research_20250302_1430_iphone16-pro评测/`

### 文件夹结构

```
research_[时间]_[话题]/
├── 00_plan.md          # 研究计划（第一步完成后立即写入）
├── 01_[平台]_[序号].md  # 每条笔记/视频/网页单独一个文件
├── 02_[平台]_[序号].md
├── ...
└── summary.md          # 最终综合报告
```

### 单条内容文件格式（`[平台]_[序号].md`）

```markdown
# [标题 / 视频名 / 帖子标题]

## 基本信息
- **平台**：小红书 / B站 / 微博 / Twitter / 抖音
- **来源链接**：[原始 URL]
- **作者/账号**：[名称]
- **发布时间**：[日期，如无则标注「未知」]
- **内容类型**：图文 / 视频 / 推文 / 长文

## 内容摘要
[100-300字，核心观点或内容总结。若调用了 getnote-video-summary，此处填写转写摘要]

## 可信度评估
- **可信度**：高 / 中 / 低
- **评估依据**：[例：账号有认证、数据有来源 / 纯主观感受、无佐证 / 明显情绪化]

## 广告/推广风险
- **可能是广告**：是 / 否 / 疑似
- **判断依据**：[例：带有品牌合作标注 / 用词过于正面且缺乏缺点 / 评论区有引流链接]

## 备注
[其他值得注意的点，例如：评论区反馈、数据异常、内容明显受审查影响等]
```

### 计划文件格式（`00_plan.md`）

- 微博：[角度]

[是否计划调用 getnote-video-summary，哪些平台可能需要]

## 工作流程（Plan Mode）

### 第一步：输出研究计划（必须，不可跳过）

收到研究请求后，**先不执行搜索**，输出结构化计划并**同步写入 `00_plan.md`**：

```
## 📋 研究计划

**研究话题**：[话题]
**搜索风格**：[日常 / 学术]

**平台分工**：
- 小红书：[计划搜索的具体角度]
- B站：[计划搜索的具体角度]
- 微博：[...]
- Twitter/X：[...]
- 抖音：[...]（如不适用说明原因）

**预计子问题**：
1. [子问题 1]
2. [子问题 2]

**视频转写**：[是否计划调用 getnote-video-summary]

---
确认开始研究？还是需要调整方向？
```

等用户确认后进入第二步。

### 第二步：多平台搜索 + 实时存档

⚠️ **重要：必须直接用浏览器打开平台搜索！不要只用 Brave/Google 搜索！**

按计划逐平台搜索，**每找到一条值得记录的内容，立即创建对应的 `.md` 文件**，不要等全部搜索完再统一写。

**正确的搜索方式**：
1. 用 `browser` 工具直接打开目标平台搜索页面（如小红书、微博、抖音、B站）
2. 用平台内置搜索框输入关键词
3. 浏览搜索结果，提取关键信息

**为什么不能用 API 搜索代替**：
- 很多平台（小红书、微博、抖音）的内部内容无法被 Google/Bing 索引
- 只有直接打开平台网站才能获取最新、最准确的信息
- 网感内容、热点梗、黑话等必须在平台上搜

**搜索策略**：
- 同一话题用平台语言习惯搜索（小红书用「推荐」「测评」「值得买吗」，Twitter 用英文专业词汇）
- 如果某平台结果稀少或质量低，记录原因后跳过，不要硬凑
- 优先近 3 个月的内容，时效性敏感话题优先近 1 个月
- 善用平台的「热门搜索」「大家都在搜」等功能发现新线索

**存档命名示例**：
- `xiaohongshu_001.md`、`xiaohongshu_002.md`
- `bilibili_001.md`
- `twitter_001.md`

### 第三步：整理原始结果（对话中呈现）

对话里每个平台单独呈现：

```
### 📱 [平台名]

**搜索词**：[实际使用的关键词]
**收录条数**：[N 条，已存入文件夹]
**结果摘要**：
- [[文件名] 作者：核心观点]（可信度：高/中/低，广告风险：低/疑似）
- ...

**平台整体倾向**：[该平台上此话题的主流声音]
```

### 第四步：综合评论 + 写入 summary.md

对话输出综合报告，**同步写入 `summary.md`**：

```
## 🔍 综合评论

**跨平台共识**：[各平台都在说什么]

**平台分歧**：[哪些平台观点不同，可能的原因]

**信息盲区**：[哪些角度目前搜不到]

**可信度总览**：[高可信度来源有哪些，哪些平台整体水分较大]

**建议后续行动**：[如果用户想深入，下一步可以怎么做]
```

## 边界情况处理

- **话题在某平台明显不适合**：直接跳过该平台并说明，不要浪费搜索次数
- **涉及敏感内容**：如果某平台的内容受审查影响，主动说明信息可能不完整
- **英文话题**：日常模式下仍然用中文总结，但原始引用可保留英文；学术模式下可以双语
- **用户只想看某个平台**：尊重用户要求，不强制走完所有平台

## 常见陷阱与教训

### 🚫 谐音梗/网络用语不要瞎猜

**案例：「逼连逼」研究错误**
- 错误：看到"逼连逼"谐音像"Kick Flip"，就武断认为是 KICKFLIP 男团
- 实际：是 SM 新女团 Hearts2Hearts 的黑称

**教训：**
1. **谐音只是线索，不是证据** - 必须看粉丝实际怎么用这个词
2. **交叉验证** - 搜「关键词+候选答案」组合（如"逼连逼 Hearts2Hearts"）确认关联
3. **核实歌曲/人名** - 看到歌曲名或人名与团体关联时，必须单独验证
4. **不确定时不瞎填** - 可以标注"推测为XX，需进一步验证"

### 🚫 避免过度自信

- 看到部分信息就想当然补全
- 急于下结论而不深挖
- 忽略反证信息

**正确做法：**
- 每个结论都要有明确来源支撑
- 存疑的地方标注出来
- 主动寻找反证，而非只找支持自己假设的证据

## 语言规范

- 默认输出语言：**中文**（简体）
- 引用原文时可保留原语言（英文推文不翻译，保留原味）
- 避免机械翻译，尤其是平台特有的网络用语（「绝绝子」「yyds」等）应保留并适当解释
- 学术模式下语气正式，避免网络用语
